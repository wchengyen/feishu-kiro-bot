A bridge service that connects the Feishu (Lark) messaging platform to the [Kiro CLI](https://kiro.dev) AI agent framework. Users interact with Kiro's capabilities—such as infrastructure analysis, report generation, and automated troubleshooting—directly through a Feishu chat interface.

**No public IP required. No port forwarding. No nginx reverse proxy.**

## ✨ Core Features

| Feature | Description |
|---------|-------------|
| 🔌 **WebSocket Long-Connection** | Outbound connection, natively穿透s NAT/firewall, zero network config |
| 🧠 **Dual-Layer Memory** | Semantic Memory (user preferences) + Episodic Memory (system events), SQLite-based |
| ⚡ **Hybrid Execution Engine** | Sync 120s → auto background async with progress heartbeats |
| 🗂️ **Session Routing** | 30min auto-resume, supports `/new` `/resume` `/sessions` |
| 📎 **Auto Resource Upload** | Detects image/file paths in Kiro output, uploads to Feishu automatically |
| ⏰ **Scheduled Tasks** | Natural language cron jobs, `/schedule` command management |
| 📝 **Event Ingestion** | `/event` manual entry + Webhook external system push |
| 🚨 **EC2 Alert Analysis** | Prometheus/CloudWatch alerts auto-trigger Kiro Skill analysis + Feishu push |

## 🚀 Quick Start

### 1. Feishu Open Platform Setup

1. Visit https://open.feishu.cn/app and log in
2. Create a custom app, record **App ID** and **App Secret**
3. Add "Bot" capability
4. Event Subscription → Choose **"Use Long Connection"** → Add `im.message.receive_v1`
5. Permissions → Enable `im:message`, `im:message:send_as_bot`, `im:resource`
6. Version Management → Submit for review → Publish

> Full permissions in `feishu-auth.json`. Minimum: `im:message` + `im:message:send_as_bot`.

### 2. Configure Service

```bash
cd /home/ubuntu/kiro-devops
cp .env.example .env
# Edit .env, fill in FEISHU_APP_ID and FEISHU_APP_SECRET
```

**Optional Config:**

| Variable | Description | Default |
|----------|-------------|---------|
| `KIRO_TIMEOUT` | Kiro CLI sync timeout (s) | `120` |
| `KIRO_ASYNC_TIMEOUT` | Async task max wait (s) | `1800` |
| `KIRO_AGENT` | Kiro agent name | empty |
| `ENABLE_MEMORY` | Enable memory feature | `false` |

### 3. Start Service

```bash
# Foreground debug
./start.sh

# systemd background (production)
sudo cp kiro-devops.service /etc/systemd/system/
sudo systemctl daemon-reload
sudo systemctl enable kiro-devops
sudo systemctl start kiro-devops
```

### 4. View Logs

```bash
sudo journalctl -u kiro-devops -f
```

## 🧠 Memory System

Memory is disabled by default. Set `ENABLE_MEMORY=true` in `.env` to enable.

**Zero additional dependencies**, all powered by Python built-in `sqlite3`.

### Dual-Layer Memory

| Type | Content | Retrieval | Prompt Injection |
|------|---------|-----------|-----------------|
| **Semantic** | User preferences, facts, decisions | Keyword overlap scoring | Prefix injection, affects Agent behavior |
| **Episodic** | System changes, deploys, alerts, incidents | Time + Entity + Type filter | Appended after user prompt, marked "for reference only" |

### Event Ingestion

**Method 1: Manual via Feishu**

```
/event type=系统变更 entities=test1,MySQL title="test1 index optimization" description="added composite index"
```

**Method 2: External Webhook Push**

```json
{
  "id": "jenkins-12345",
  "event_type": "应用发版",
  "title": "Order Service v2.3.1 deployed",
  "description": "fixed payment callback timeout",
  "entities": ["order-service"],
  "source": "jenkins",
  "user_id": "ou_xxx"
}
```

### Memory Management Commands

| Command | Function |
|---------|----------|
| `/memory status` | Show memory state and semantic memory count |
| `/memory on` | Enable memory |
| `/memory off` | Disable memory |
| `/memory clear` | Clear semantic memories |
| `/memory events` | List recent 30-day events |
| `/memory events clear` | Clear all event records |

## 🚨 EC2 Alert Auto-Analysis (Webhook + Kiro Skill)

Receive EC2 alerts from Prometheus/CloudWatch, automatically trigger the Kiro `ec2-alert-analyzer` skill for root cause analysis, and proactively push results to Feishu.

### Architecture

```
┌─────────────────┐     webhook POST      ┌─────────────────┐
│  Prometheus     │ ─────────────────────>│                 │
│  Alertmanager   │   Bearer Token + JSON │  kiro-devops│
│  (or CloudWatch │                       │   :8080/event   │
│   via Lambda)   │                       │                 │
└─────────────────┘                       └────────┬────────┘
                                                   │
                              ┌────────────────────┼────────────────────┐
                              ↓                    ↓                    ↓
                       ┌────────────┐      ┌────────────┐      ┌────────────┐
                       │ Persist    │      │ severity   │      │ Low-sev    │
                       │ to events  │      │ >= high?   │      │ alert only │
                       │   .db      │      └─────┬──────┘      │ persisted  │
                       └────────────┘            │              └────────────┘
                                                 ↓ Yes
                                    ┌────────────────────────┐
                                    │ Kiro ec2-alert-analyzer│
                                    │   skill auto-queries   │
                                    │  • CloudWatch CLI      │
                                    │  • Prometheus API      │
                                    │  • Root cause analysis │
                                    └───────────┬────────────┘
                                                │
                                                ↓ Result
                                    ┌────────────────────────┐
                                    │  send_message()        │
                                    │  Push to Feishu user   │
                                    └────────────────────────┘
```

**Design principle**: Bot acts only as a gateway (receive → forward to Kiro → push result). All analysis logic lives in the Kiro Skill.

### Configuration

#### 1. Environment Variables (`.env`)

```bash
# === Webhook Alert Receiver ===
WEBHOOK_ENABLED=true
WEBHOOK_PORT=8080
WEBHOOK_HOST=127.0.0.1          # 127.0.0.1=localhost only, 0.0.0.0=all interfaces
WEBHOOK_TOKEN=change-me-secret

# === Proactive Alert Push ===
ALERT_NOTIFY_USER_ID=ou_xxxxxxxxxxxxxxxx    # Feishu open_id
ALERT_AUTO_ANALYZE_SEVERITY=high,critical   # Severities that trigger auto-analysis
ALERT_ANALYZE_TIMEOUT=300                   # Kiro analysis timeout (seconds)
```

#### 2. Prometheus Alertmanager Config

```yaml
# alertmanager.yml
global:
  resolve_timeout: 5m

route:
  group_by: ['alertname', 'instance']
  group_wait: 10s
  group_interval: 10s
  repeat_interval: 1h
  receiver: 'kiro-devops'

receivers:
  - name: 'kiro-devops'
    webhook_configs:
      - url: 'http://bot.internal:8080/event'
        http_config:
          bearer_token: 'change-me-secret'
        send_resolved: true
```

Alertmanager's native JSON payload is **automatically detected and converted** to the Bot's standard format—no extra adapter needed.

#### 3. CloudWatch Relay (Lambda)

CloudWatch Alarm → SNS → Lambda converts to Bot standard JSON. See `.kiro/skills/ec2-alert-analyzer/SKILL.md` for details.

### Severity-Based Response

| Severity | Behavior |
|----------|----------|
| `critical` / `high` | Auto-trigger Kiro skill analysis + proactive Feishu push |
| `medium` / `low` | Persisted only; user can query later |

### Test Verification

```bash
# 1. Health check
curl http://localhost:8080/health
# {"status": "ok", "event_store": true, "webhook": true}

# 2. Simulate Prometheus Critical alert (triggers auto-analysis)
curl -X POST http://localhost:8080/event \
  -H "Authorization: Bearer change-me-secret" \
  -H "Content-Type: application/json" \
  -d '{
    "id": "prom-ec2-cpu-001",
    "event_type": "指标异常",
    "title": "test1 EC2 CPU usage > 80%",
    "description": "CPU utilization is 85.2%",
    "entities": ["test1", "i-0abcd1234"],
    "source": "prometheus",
    "severity": "critical",
    "timestamp": "2026-04-23T10:00:00Z"
  }'
# {"ok": true, "event_id": "prom-ec2-cpu-001", "analysis_triggered": true}

# 3. Alertmanager native format
curl -X POST http://localhost:8080/event \
  -H "Authorization: Bearer change-me-secret" \
  -d '{"version":"4","status":"firing","commonLabels":{"alertname":"HighMemoryUsage","instance":"test2:9100","severity":"high"},"commonAnnotations":{"summary":"test2 memory usage > 90%"},"alerts":[{"status":"firing","startsAt":"2026-04-23T11:00:00.000Z"}]}'
# {"ok": true, "event_id": "prom-HighMemoryUsage-...", "analysis_triggered": true}

# 4. Low severity (no analysis triggered)
curl ... -d '{"id":"prom-disk-low-001","severity":"low","title":"disk 60%"}'
# {"ok": true, "analysis_triggered": false}

# 5. Auth failure
curl -H "Authorization: Bearer wrong-token" ...
# 401 Unauthorized

# 6. Idempotent retry
curl ... -d '{"id":"same-id","severity":"low"}'  # First: persisted
curl ... -d '{"id":"same-id","severity":"low"}'  # Second: skipped (no dup)
```

### Kiro Skill Auto-Analysis Example

When receiving `test1 EC2 CPU usage > 80%`, the Kiro `ec2-alert-analyzer` skill autonomously executes:

```bash
# Query CloudWatch CPU trend
aws cloudwatch get-metric-statistics \
  --namespace AWS/EC2 --metric-name CPUUtilization \
  --dimensions Name=InstanceId,Value=i-0abcd1234 \
  --start-time ... --end-time ... --period 300 --statistics Average Maximum

# Or query Prometheus node_exporter
curl -s 'http://prometheus:9090/api/v1/query?query=100-avg(irate(node_cpu_seconds_total{mode="idle",instance=~"test1:9100"}[5m]))*100'
```

Output is a structured Chinese report with root cause, impact assessment, and actionable recommendations.

## 💬 Multi-Turn Conversations

### Automatic Session Continuation

By default, the bot automatically continues the context of the same topic. If you send another message **within 30 minutes**, it will auto-resume the same session, and Kiro CLI will carry the full conversation history for reasoning.

### Explicit Session Management

When you need to switch topics, use these commands:

| Command | Description |
|---------|-------------|
| `/new` | Force start a new session; next message will not be affected by historical context |
| `/sessions` | View the 10 most recent historical sessions |
| `/resume <id>` | Resume a specific historical session and continue the previous conversation |

> 💡 **Tip**: If the bot's reply drifts off-topic (e.g., resumed an old session), send `/new` to reset.

## ⏰ Scheduled Tasks

Configure periodic tasks via natural language. The bot will automatically execute Kiro commands at the specified time and push results to you.

**Usage examples:**
```
/schedule check AWS cost every day at 9am
/schedule backup database every Monday at 2am
/schedule check EC2 instance status every 30 minutes
```

**Management commands:**
```
/schedule list      # List all scheduled tasks
/schedule delete 1  # Delete task #1
/schedule help      # Show help
```

## 📎 Image & File Delivery

The bot supports automatic detection of file paths in Kiro output and uploads them to Feishu:

- **Images**: `.png` `.jpg` `.jpeg` `.gif` `.bmp` `.webp` → sent as image messages
- **Files**: `.pdf` `.doc` `.docx` `.xls` `.xlsx` `.ppt` `.pptx` `.csv` `.txt` `.zip` `.mp4` → sent as file messages

**Workflow:**
1. You send a request to the bot (e.g., "generate a CPU trend chart")
2. Kiro processes and generates a file, outputting an absolute path (e.g., `/tmp/report/cpu.png`)
3. The bot automatically detects the existing file path
4. Uploads to Feishu and replies as an image/file message

> **Note**: Requires `im:resource` permission in the Feishu Open Platform (for uploading images/files).
